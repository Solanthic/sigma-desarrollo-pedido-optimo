# Computation Graph: Raw vs Derived Data & Embedded Business Logic

This document traces every table in the Pedido Optimo pipeline to its **origin** — whether it's raw source data or a derived/calculated artifact — and documents the **exact business logic** embedded in each computation. The goal is to prevent blindspots when migrating to Foundry: if we just import a "result" table, we lose the logic that created it and can't recalculate, debug, or scale it.

---

## 2. The Core Business Logic (Exact Formulas)

### 2.1 Inventario Sugerido (Suggested Inventory Target)

**Location**: `my_query1.sql` (lines in CASE statement) + used in NB01

**Formula**: `Inventario_sugerido = (Scan_pizas / 7) * days_multiplier`

Where `days_multiplier` depends on `Tipo_merma`:

| Tipo_merma | Days Multiplier | Meaning |
|---|---|---|
| `Ok` | 14 | Low shrinkage — carry 14 days of supply |
| `Alta` | 12 | Elevated shrinkage — carry 12 days |
| `Muy Alta` | 10 | High shrinkage — carry 10 days |
| `Scritica` | 8 | Sub-critical — carry 8 days |
| `Critica` | 7 | Critical shrinkage — carry 7 days |
| `Inconsistente` | 0 | Unpredictable — don't stock |
| NULL | 0 | No classification — triggers reactivation |

**Inputs needed**:
- `Scan_pizas` — weekly average scanned sales in pieces (from `Venta_scan_semanal_prom`, T07)
- `Tipo_merma` — shrinkage classification (from `mermas_autos_test_pedido_sugerido`, T05)

**BLINDSPOT**: We do NOT know how `Tipo_merma` is classified. The classification logic is upstream of our pipeline. This is the most important variable in the entire system — it determines how much inventory to target — and we cannot reproduce it.

**BLINDSPOT**: We do NOT know how `Scan_pizas` is aggregated. It's a weekly average, but over what time window? How are outliers handled? Is it rolling 4 weeks? 8 weeks?

### 2.2 OH_Piezas (On-Hand Pieces)

**Location**: `my_query1.sql`

```sql
CASE 
  WHEN Presentacion = 'GRANEL' THEN ROUND(OH_kilos / Peso, 2)
  ELSE ROUND(OH_kilos / Peso, 0)
END AS OH_Piezas
```

**Logic**: Convert kilos to pieces using SKU weight. Bulk (GRANEL) items get 2 decimal places; packaged items are rounded to whole numbers.

**Inputs**: `OH_kilos` (T01), `Peso` (T03), `Presentacion` (T03)

### 2.3 OH_proyectado (Projected Inventory)

**Location**: NB01 (Python)

```python
OH_proyectado = round(OH_Piezas.fillna(0) + Piezas_transito.fillna(0), 0)
```

**Logic**: Current inventory + incoming transit = what we'll have when transit arrives.

**Inputs**: `OH_Piezas` (computed above), `Piezas_transito` (from transitos.csv, T19)

### 2.4 Piezas_empuje (Push Quantity — The Optimal Order)

**Location**: NB01 (Python)

```python
Piezas_empuje = WHERE(
    (Tipo_merma == 'Ok') AND (Inventario_sugerido > OH_proyectado),
    round(Inventario_sugerido - OH_proyectado, 0),
    0
)
```

**Logic**: Only push when:
1. Shrinkage is classified as `Ok` (low risk)
2. Target inventory > projected inventory (there's a gap to fill)

Push quantity = target - projected = how many units to order.

**IMPORTANT**: Push is ONLY triggered for `Ok` merma. All other classifications (Alta, Muy Alta, etc.) produce Piezas_empuje = 0, even if there's a gap. This is a conservative business rule — only push product with low waste risk.

### 2.5 Q-FRESCOS Discount

**Location**: NB01 (Python)

```python
IF Piezas_empuje > 0 AND linea == 'Q-FRESCOS':
    Piezas_empuje = round(Piezas_empuje * 0.33, 0)
```

**Logic**: Fresh products get a 67% reduction in push quantity (multiply by 0.33). Business reason: higher spoilage risk for fresh items, so be conservative.

### 2.6 No_cargar_pedido (Cut Flag — Don't Order)

**Location**: NB01 (Python)

```python
No_cargar_pedido = WHERE(
    (Tipo_merma NOT IN ('Ok', 'Alta')) AND (OH_proyectado > Inventario_sugerido),
    1, 0
)
```

**Logic**: Flag = 1 when high-shrinkage products already have MORE inventory than target. Signal to cut (reduce) the order, not increase it.

### 2.7 Priority Scoring

**Location**: NB01 (Python)

| Condition | Priority | Meaning |
|---|---|---|
| Piezas_empuje > 0 AND Top_venta = 1 | **1** | Push opportunity + top seller — highest priority |
| Piezas_empuje > 0 AND Top_venta = 0 | **2** | Push opportunity + non-top seller |
| Otherwise | **0** | No push opportunity |

### 2.8 Empuje_cabecera (Promotional Push)

**Location**: NB01 (Python)

```python
Empuje_cabecera = WHERE(
    (Objetivo_cabecera - OH_proyectado > 0) AND (Con_cabecera_activa == 1),
    Objetivo_cabecera - OH_proyectado,
    0
)
```

**Logic**: For active promotions, push to the promotional objective. Independent of regular push logic.

### 2.9 Reactivation Flags

**Location**: `my_query1.sql`

```sql
Reactivar_con_pedido_operacion = CASE WHEN Tipo_merma IS NULL AND operacion = 1 THEN 1 ELSE 0 END
Reactivar_con_pedido_central   = CASE WHEN Tipo_merma IS NULL AND operacion = 0 THEN 1 ELSE 0 END
```

**Logic**: When a store-SKU has no shrinkage classification (NULL Tipo_merma) but IS in the active catalog, it needs reactivation. The `operacion` flag determines who should reactivate it.

### 2.10 Wal-Mart GRANEL Exclusion

**Location**: NB04 (Python)

```python
EXCLUDE WHERE (Grupo == 'Wal-Mart') AND (Presentacion == 'GRANEL')
```

**Logic**: Never push bulk items to Wal-Mart stores. Business rule — Wal-Mart handles bulk differently.

### BLINDSPOT 1: `Tipo_merma` Classification (CRITICAL)

**Values**: Ok, Alta, Muy Alta, Scritica, Critica, Inconsistente

**Where it lives**: `mermas_autos_test_pedido_sugerido` table (T05). In Snowflake, embedded in `SOP_PAC_PEDIDO_BASE`.

**What we now know** (from commercial-strategy context):
- Shrink/waste classes are part of a **safety stock rule system** that considers product category, seasonality, price effects, and category-specific exceptions
- "Phantom inventory" (ghost inventory) is detected as >21 days without sales — triggers exclusion from ordering. Future improvement: move to SKU-specific cadence-aware thresholds
- The classification likely comes from the ~1,400-line SQL process that runs in ~2 hours (Walmart-specific, mentioned in March 17 meeting)
- Innovation products and short-shelf-life autoservices items need separate classification rules

**What we still DON'T know**:
- The exact thresholds or model that maps raw shrinkage data → category labels (Ok/Alta/etc.)
- Whether the classification is statistical, rule-based, or hybrid
- How often it's recalculated
- Who maintains the rules — Enrique Morales (leaving 2026-05-04)?

**Risk**: If we don't know how to classify Tipo_merma, we can use historical values but cannot extend to new store-SKUs, new regions, or recover from stale classifications.

**Action needed**: Ask Enrique Morales for the classification logic before his transition date (2026-05-04).

### BLINDSPOT 2: `Scan_pizas` / Weekly Sales Velocity (HIGH)

**What**: The weekly average scanned sales — the denominator of the target inventory formula.

**Where it lives**: `Venta_scan_semanal_prom` table (T07). In Snowflake, SCAN_PROMEDIO_PIEZAS in `SOP_PAC_PEDIDO_BASE`.

**What we now know** (from commercial-strategy context):
- Source is Pavis scanner-sales feed (retailer POS data)
- The system also uses Retail Link extracts (Walmart-specific) and a `Historico` table for velocity/trend analysis
- Sales velocity is used for: projected inventory decay, phantom inventory detection, safety-stock levels, and anomaly flagging
- `TBL_RM_CTX` (664M rows in Snowflake, NOT synced to Foundry) likely contains the raw daily scan data from which this is aggregated

**UPDATE (2026-04-06) — Partially resolved**: `sell_out_oh_diarios_28` schema confirmed — contains `Scan_kilos` at store-SKU-day grain alongside `OH_kilos`. The raw daily data EXISTS and `Scan_pizas` = `SUM(Scan_kilos) / Peso` over some window. The "28" in the name suggests a 28-day (4-week) rolling window. We can now recompute this in Foundry.

**What we still DON'T know**:
- ~~Over what time window?~~ Likely 28 days (4 weeks) based on dataset name
- Are outliers or promotions excluded?
- Is it a simple mean or weighted (more recent = more weight)?

**Risk**: Reduced — we now have the raw daily data and can experiment with different aggregation windows in Foundry.

### BLINDSPOT 3: `Inventario_optimo` in TC_RL Table (MEDIUM)

**What**: A **different** optimal inventory model that produces `Inventario_optimo`, `DOH_Actuales`, `Pedido_minimo_sugerido`.

**Where it lives**: `mermas_autos_TC_Inventario_optimo` (T16) in `indicadoresdb`. In Snowflake: `SOP_PAC_TC_RL` (95 rows, schema validated).

**What we now know**:
- TC_RL only covers `Clase = 'RL'` products (95 rows in Pachuca)
- It already includes `PIEZAS_TRANSITO` — transit integration is baked in
- The schema matches `my_query.sql` output exactly, confirming it's the output of the standalone RL-class report
- It applies additional filters vs NB01: excludes GRANEL, Critica, Inconsistente, and suggested orders ≤ 5 pieces

**What we still DON'T know**:
- How is `Inventario_optimo` calculated? Different from `Inventario_sugerido` in NB01
- Is TC_RL a newer/better model that should replace NB01's logic?
- Only covers RL class — what about other classes?

**Risk**: Two competing "optimal inventory" calculations exist. Which one is correct? If we build Foundry around the wrong one, recommendations will diverge from what operations expects.

## 4. Web App Role: What's Calculated vs What's Captured

The MySQL/PHP web app on HostGator (`soporteracu.com`) is NOT a raw data source — it's a **bidirectional interface** where:

### What the Team **Uploads TO** the Web App (calculated data):
- Recommended/suggested orders (output of the NB01-NB02 pipeline)
- Projected inventory positions
- Push/cut alerts and reasoning
- These flow from SQL Server → pipeline computation → web app database

### What Operations **Inputs INTO** the Web App (captured data):
- **Real inventory readings** — promoters count actual store inventory; this is the highest-confidence inventory source and overrides system theoretical levels
- **Notes and comments** — operational context from field staff
- **Order adjustments** — store/field people can **modify the suggested order** quantities with justification before the SAP load cutoff

### What This Means for the Computation Graph:
- `proyeccion_detalle_pedidos` (T18, now in Snowflake with 108K rows) contains the **adjusted/final orders** — these are post-human-review, not purely algorithm-generated
- `SOP_PAC_MODIFICADO` (28,795 rows) in Snowflake likely contains the modification records — the delta between suggested and actual
- `SOP_PAC_COMENTARIOS` (15 rows) contains operational notes
- The promoter-captured inventory counts are the **preferred source** for `OH_Piezas` — the pipeline should track `inventory_source` (promoter_count vs system_theoretical) to preserve this distinction

### Implication for Foundry:
The web app's execution/feedback loop (recommend → review → adjust → load → deliver → reconcile) maps directly to pipeline stages T7-T10 in the target architecture. In Foundry, this becomes:
- `OptimalOrderDecision` → `OrderAdjustmentRequest` → `OrderLoadResult` → `DeliveryOutcome`
- The adjust step is where human judgment modifies algorithm output — critical to preserve and audit

## 9. Foundry Rebuild Strategy Implications

When rebuilding in Foundry, we need to decide for each derived table:

1. **Import the pre-computed result** — fast but fragile, loses logic, can't scale
2. **Replicate the computation in Foundry** — requires understanding the source logic
3. **Replace with a better model** — opportunity to improve, but need to validate against current outputs

| Table | Recommendation | Rationale |
|---|---|---|
| T05 (`pedido_sugerido`) | **Must understand logic first** | Tipo_merma classification is the most important variable. Can't replicate or replace without knowing how it works. |
| T07 (`venta_scan`) | **Replicate in Foundry** | If we get daily scan data (likely in Pavis/TBL_RM_CTX), we can compute the weekly average ourselves with a Foundry transform. |
| T08 (`sku_insignia`) | **Replicate in Foundry** | Top-seller flag can be computed from sales data if we have the ranking criteria. |
| T16 (`TC_Inventario_optimo`) | **Understand first, then decide** | Two competing models exist. Need to pick one or merge them. |
| NB01 logic | **Replicate in Foundry transform** | All formulas documented above. Straightforward to implement as a Python or SQL transform. |
| NB02 logic | **Replicate in Foundry** | Simple merge + day-of-week filter. |
| NB03-05 logic | **Replace with Foundry actions/notifications** | PDF generation and SFTP upload should use Foundry's native capabilities. |
